# Teaching a Neural Network to Learn

*Project 2 in my build series at github.com/dwinsi/LLMfromScratch*

In Project 1, I built a single neuron that decided whether to bring an umbrella. It worked. But the weights were picked by me. I chose 0.6 for cloud cover and 0.3 for humidity because they made intuitive sense. The neuron had no say in it.

That is not a neural network that learns. That is me doing the learning and writing the results into a script.

Project 2 changes that. The network picks its own weights from examples. I give it labelled data and it figures out what the weights should be by itself.

These are my notes on how that actually works.

---

## What changes in Project 2

Three things are new here that were not in Project 1.

First, the network now has multiple neurons arranged in layers. An input layer, a hidden layer and an output layer. The hidden layer is where the network learns to combine features in ways I would never think to specify by hand.

Second, the network needs a way to measure how wrong it is at any given moment. That measurement is called the loss. Without it, the network has no signal to learn from.

Third, the network uses that loss measurement to adjust its weights in the right direction. This process is called backpropagation. It is the core of how every neural network learns, from the smallest toy example to the largest LLM.

---

## The training data

Still the umbrella problem. Six examples of weather conditions with labels telling the network whether an umbrella was the right call.

```
cloud_cover  humidity  wind_speed  ->  bring_umbrella
    0.9         0.7       0.3              1  (yes)
    0.1         0.2       0.1              0  (no)
    0.8         0.6       0.5              1  (yes)
    0.2         0.3       0.2              0  (no)
    0.7         0.8       0.4              1  (yes)
    0.1         0.1       0.9              0  (no)
```

Six examples is a tiny dataset. In a real network you would use thousands. But six is enough to see the learning happen clearly.

---

## The network architecture

The network has three layers.

The input layer has three neurons, one for each feature: cloud cover, humidity and wind speed. This layer does no computation. It just holds the input values.

The hidden layer has four neurons. Each one receives all three inputs, applies its own weights, and passes its output forward. This is where the interesting learning happens.

The output layer has one neuron. It receives the four outputs from the hidden layer and produces a single number between 0 and 1, the predicted probability of needing an umbrella.

In matrix terms:

```
training_inputs:          shape (6, 3)   -> 6 examples, 3 features each
weights_input_to_hidden:  shape (3, 4)   -> connects 3 inputs to 4 hidden neurons
bias_hidden:              shape (1, 4)
hidden_layer_output:      shape (6, 4)   -> 4 hidden outputs for each of 6 examples
weights_hidden_to_output: shape (4, 1)   -> connects 4 hidden neurons to 1 output
bias_output:              shape (1, 1)
predicted_output:         shape (6, 1)   -> 1 prediction for each of 6 examples
```

At the start, both weight matrices are filled with small random numbers. The network knows nothing yet.

## Why 4 hidden neurons?

I chose 4 and it is a fair question to ask how.

There is no formula for this. The number of hidden neurons is a decision guided by intuition, heuristics and trial and error. It is one of those things practitioners learn by building many networks rather than deriving from first principles.

A few rough guides that help in practice.

**Input and output size as a starting point.** The hidden layer usually sits somewhere between the size of the input and the size of the output. Our input is 3 features and our output is 1 number. So anywhere from 2 to 8 hidden neurons is a reasonable starting range for a problem this simple.

**Underfitting vs overfitting.** Too few hidden neurons and the network cannot learn the pattern at all. This is called underfitting. Too many and it memorises the training examples rather than learning a general rule. This is called overfitting. With only 6 training examples like ours, 4 hidden neurons is already at risk of overfitting. But the dataset is so small it does not matter for the teaching purpose. In a real project with thousands of examples, this balance becomes one of the most important decisions in the whole design.

**Computational budget.** More neurons mean more parameters, more matrix operations, more compute, and slower training. For a toy example like this one, it is completely irrelevant. For a real network training on millions of examples over many hours or days, the number of hidden neurons in each layer has a direct cost attached to it.

**Powers of 2.** In practice most practitioners use 32, 64, 128, 256 or similar numbers for hidden layer sizes. Modern hardware, especially GPUs, is optimised for matrix operations on powers of 2, so these sizes tend to train faster than arbitrary numbers like 37 or 91. This is a practical convention, not a mathematical rule.

For now, 4 works, 2 would also work, and the honest answer is that I tried a few values and this one trained cleanly. Choosing architecture hyperparameters well is a skill that comes with experience. There is no shame in starting with a reasonable guess and adjusting when it does not work.

## The forward pass

The forward pass is the same idea as Project 1, just applied twice in sequence.

**Layer 1.**

```
hidden_layer_input  = training_inputs · weights_input_to_hidden + bias_hidden
hidden_layer_output = sigmoid(hidden_layer_input)
```

Each of the six input examples gets multiplied by weights_input_to_hidden, the bias is added, and sigmoid squashes the result. The output hidden_layer_output is a matrix of shape (6, 4). Six examples, four hidden neuron outputs each.

**Layer 2.**

```
output_layer_input = hidden_layer_output · weights_hidden_to_output + bias_output
predicted_output   = sigmoid(output_layer_input)
```

The hidden layer outputs go through the same process with weights_hidden_to_output. The final predicted_output is shape (6, 1). One prediction per example.

At the very start, with random weights, these predictions are essentially random too. Maybe 0.48, 0.51, 0.53. The network is guessing.

![Forward pass flow showing data moving through the two layers](/02-network/images/forward_pass.png)

## The loss function

To teach the network anything, I need a number that tells it how wrong it is. That number is the loss.

For this project I use mean squared error. The formula is simple.

```
loss = mean( (actual - predicted)² )
```

For each example, I take the difference between the actual label and the predicted output, square it (to make all values positive), and average across all examples.

If all predictions are perfect, loss is 0. If predictions are terrible, loss is large.

On the first pass with random weights:

```
initial loss = 0.3271
```

That is the starting point. The network is quite wrong. Every training step from here is aimed at reducing this number.

---

## Backpropagation: the intuition

Before the math, here is the idea.

Imagine the loss is a hill. The network's weights are its position on that hill. The goal is to walk downhill until reaching the lowest point, which is the minimum loss.

To walk downhill, the network needs to know which direction is downhill. That is exactly what the derivative tells us. The derivative of the loss with respect to each weight says: if I increase this weight slightly, does the loss go up or down? If it goes up, decrease the weight. If it goes down, increase it. This process is called gradient descent.

Backpropagation is how the derivatives are computed. It works by applying the chain rule backwards through every layer, from the output back to the inputs.

---

## Backpropagation: the math

This is where the sigmoid derivative we saved from Project 1 finally appears.

The derivative of sigmoid is elegant. If the sigmoid output is s, the derivative is:

```
sigmoid'(x) = s × (1 - s)
```

This says: how sensitive is the sigmoid output to a small change in its input. When s is close to 0.5, the answer is 0.25, quite sensitive. When s is close to 0 or 1, the answer is nearly 0, barely sensitive at all. This is the saturation problem we noted at the edges of the table in Project 1.

![Sigmoid curve alongside its derivative showing the saturation at the edges](/02-network/images/sigmoid_derivative.png)

Now the chain rule, applied backwards through the network.

**Step 1: how does loss change with respect to predicted output?**

```
gradient_loss_wrt_predicted = -2 × (training_labels - predicted_output) / n
```

Where n is the number of examples. This is just the derivative of mean squared error.

**Step 2: how does predicted output change with respect to output layer input?**

```
gradient_predicted_wrt_output_input = sigmoid'(output_layer_input)
                                     = predicted_output × (1 - predicted_output)
```

The sigmoid derivative evaluated at the output layer.

**Step 3: combine into one output delta.**

```
gradient_output_delta = gradient_loss_wrt_predicted × gradient_predicted_wrt_output_input
```

**Step 4: use the output delta to get weight gradients for the output layer.**

The inputs to the output layer are the hidden layer outputs. So:

```
gradient_weights_hidden_to_output = hidden_layer_output.T · gradient_output_delta
gradient_bias_output               = sum(gradient_output_delta)
```

These tell the network exactly how to adjust the output layer weights.

**Step 5: propagate back through the hidden layer.**

```
gradient_loss_wrt_hidden_output  = gradient_output_delta · weights_hidden_to_output.T
gradient_hidden_wrt_hidden_input = sigmoid'(hidden_layer_input)
gradient_hidden_delta            = gradient_loss_wrt_hidden_output × gradient_hidden_wrt_hidden_input

gradient_weights_input_to_hidden = training_inputs.T · gradient_hidden_delta
gradient_bias_hidden             = sum(gradient_hidden_delta)
```

The chain rule has now carried the loss signal all the way back to the first layer.

**Step 6: update all weights.**

```
weights_input_to_hidden  -= learning_rate × gradient_weights_input_to_hidden
bias_hidden              -= learning_rate × gradient_bias_hidden
weights_hidden_to_output -= learning_rate × gradient_weights_hidden_to_output
bias_output              -= learning_rate × gradient_bias_output
```

The learning rate is a small number, 0.5 here, that controls how big each step downhill is. Too large and the network overshoots. Too small and training takes forever.

This entire sequence, forward pass, compute loss, backward pass, update weights, is one training epoch. We repeat it thousands of times.

## The code

```python
import numpy as np

np.random.seed(42)

def sigmoid(x):
    return 1 / (1 + np.exp(-x))

def sigmoid_derivative(x):
    sigmoid_output = sigmoid(x)
    return sigmoid_output * (1 - sigmoid_output)

# Training data: [cloud_cover, humidity, wind_speed] -> bring_umbrella
training_inputs = np.array([
    [0.9, 0.7, 0.3],   # cloudy, humid, light wind -> yes
    [0.1, 0.2, 0.1],   # clear, dry, calm -> no
    [0.8, 0.6, 0.5],   # cloudy, humid, windy -> yes
    [0.2, 0.3, 0.2],   # mostly clear, low humidity -> no
    [0.7, 0.8, 0.4],   # cloudy, very humid -> yes
    [0.1, 0.1, 0.9],   # clear but windy -> no
])

training_labels = np.array([[1], [0], [1], [0], [1], [0]])

# Initialise weights randomly, biases at zero
weights_input_to_hidden  = np.random.randn(3, 4) * 0.5
bias_hidden              = np.zeros((1, 4))
weights_hidden_to_output = np.random.randn(4, 1) * 0.5
bias_output              = np.zeros((1, 1))

learning_rate = 0.5
epochs = 5000

for epoch in range(epochs):

    # Forward pass
    hidden_layer_input  = np.dot(training_inputs, weights_input_to_hidden) + bias_hidden
    hidden_layer_output = sigmoid(hidden_layer_input)
    output_layer_input  = np.dot(hidden_layer_output, weights_hidden_to_output) + bias_output
    predicted_output    = sigmoid(output_layer_input)

    # Loss: mean squared error
    loss = np.mean((training_labels - predicted_output) ** 2)

    # Backward pass
    gradient_loss_wrt_predicted      = -2 * (training_labels - predicted_output) / len(training_inputs)
    gradient_predicted_wrt_output_input = sigmoid_derivative(output_layer_input)
    gradient_output_delta            = gradient_loss_wrt_predicted * gradient_predicted_wrt_output_input

    gradient_weights_hidden_to_output = np.dot(hidden_layer_output.T, gradient_output_delta)
    gradient_bias_output              = np.sum(gradient_output_delta, axis=0, keepdims=True)

    gradient_loss_wrt_hidden_output  = np.dot(gradient_output_delta, weights_hidden_to_output.T)
    gradient_hidden_wrt_hidden_input = sigmoid_derivative(hidden_layer_input)
    gradient_hidden_delta            = gradient_loss_wrt_hidden_output * gradient_hidden_wrt_hidden_input

    gradient_weights_input_to_hidden = np.dot(training_inputs.T, gradient_hidden_delta)
    gradient_bias_hidden             = np.sum(gradient_hidden_delta, axis=0, keepdims=True)

    # Update weights
    weights_input_to_hidden  -= learning_rate * gradient_weights_input_to_hidden
    bias_hidden              -= learning_rate * gradient_bias_hidden
    weights_hidden_to_output -= learning_rate * gradient_weights_hidden_to_output
    bias_output              -= learning_rate * gradient_bias_output

    if epoch % 100 == 0:
        print(f"Epoch {epoch:5d}  loss: {loss:.4f}")

print()
print("Final predictions:")
for i, (predicted, actual) in enumerate(zip(predicted_output, training_labels)):
    decision = "bring umbrella" if predicted[0] > 0.5 else "leave it"
    correct  = "correct" if round(predicted[0]) == actual[0] else "wrong"
    print(f"  Sample {i+1}: predicted {predicted[0]:.4f} -> {decision} ({correct})")
```

## Adjusting training: 800 epochs vs 5000 epochs

One of the most practical things to understand about training a neural network is that you can adjust it. Running 5000 epochs is not the only option. What if 800 is enough?

The code is now structured as two experiments so you can see the difference directly.

**Experiment 1: 800 epochs**

```

**Comparison**

```

Both runs got every prediction correct. The 800 epoch run is less confident, 0.9560 instead of 0.9895 on the first sample for example, but the decisions are the same. For this simple problem, 800 epochs is already enough to get it right.

The extra 4200 epochs shaved the loss from 0.0045 to 0.0004. Whether that matters depends on the problem. For a six-example toy dataset, it does not. For a real production model, the difference between 0.0045 and 0.0004 could be significant.

This is the judgment call at the heart of training any neural network. How long is long enough? The answer is always: it depends on how much the remaining loss matters for your use case.

## The shape of learning

Looking at both runs side by side reveals something worth paying attention to.

The loss drops from 0.3271 to 0.0075 in the first 600 epochs. That is a 97.7 percent reduction. The network does the overwhelming majority of its learning almost immediately.

The remaining 4400 epochs, from epoch 600 to epoch 5000, only reduce the loss by another 0.007. That is the remaining 2.2 percent.

```
Epochs 0 to 600:    97.7% of total learning
Epochs 600 to 5000:  2.2% of total learning
```

This pattern is called diminishing returns in gradient descent. The first few steps down the hill are large because the network is far from the bottom and the gradient is steep. As it gets closer to the minimum, the steps get smaller. The hill flattens. Each additional epoch buys less improvement than the one before.

This has a practical consequence in real training. Running a model for ten times as long does not produce ten times better results. At some point, training more stops being useful. Knowing when to stop is one of the skills that comes with building many networks.

For this toy example it does not matter. But the shape of the loss curve is the same in every network, from this six-example umbrella problem to the largest language models in the world. Steep drop early, slow grind later.

## What I took away from this

Backpropagation seemed intimidating before I wrote it myself. After writing it, it is just the chain rule applied backwards through a network. Each step follows from the one before it. Each derivative has a shape that matches what you expect.

The sigmoid derivative showing up as s times one minus s is the most elegant piece of the whole thing. It means you already have everything you need. You computed sigmoid on the forward pass. On the backward pass you just use that same output again.

Nothing new is computed. The same numbers, used twice.
